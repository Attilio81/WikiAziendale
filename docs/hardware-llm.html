Use case
Cosa chiede il progetto al modello — vincoli concreti

L'agente legge procedure aziendali (2.000–5.000 token), decide se creare o aggiornare pagine wiki, chiama tool su database, produce Markdown strutturato. Cinque persone lo usano come chatbot durante la giornata lavorativa — non necessariamente tutte in simultanea, ma con picchi reali di 3–5 richieste concorrenti.

Tool calling / function calling Italiano nativo Context 128K+ Output Markdown strutturato 5 utenti concorrenti
Cambio di paradigma: da workstation a server Con un solo utente bastava LM Studio su un PC potente. Con 5 utenti serve una macchina dedicata che gira H24, espone un'API HTTP, e gestisce le richieste concorrenti. LM Studio non è adatto a questo scenario — usare Ollama o vLLM in modalità server.
Infrastruttura
Architettura per un chatbot condiviso — componenti necessari

Stack consigliato

Server → vLLM o Ollama → Open WebUI → utenti Una macchina con GPU potente gira il modello come servizio HTTP. Gli utenti accedono tramite Open WebUI (interfaccia chat web) o direttamente via API. Tutto nella rete locale — nessun dato esce dall'azienda.

Backend di inferenza: quale scegliere

vLLM
Consigliato per multi-utente
  • Batching continuo: serve più richieste in parallelo reale
  • 3–8× più throughput di Ollama con 5+ utenti
  • KV cache condivisa tra sessioni
  • Supporto tool calling nativo (OpenAI-compatible API)
  • Setup più complesso (Linux + Python)
  • CUDA 12+ richiesto, no Windows nativo
Scelta giusta per 5 utenti attivi. Richiede Linux (WSL2 su Windows è accettabile).
Ollama
Più semplice, accoda le richieste
  • Installazione in 2 minuti su Windows/Linux/Mac
  • Ottimo supporto GGUF / modelli quantizzati
  • API OpenAI-compatible pronta
  • Interfaccia Open WebUI plug-and-play
  • Processa richieste in serie (queue): latenza alta sotto carico
  • Non adatto a picchi di 5 utenti simultanei
Ok per team che usa il chatbot raramente in contemporanea. Più realistico di quanto sembri in un ufficio.

VRAM e utenti concorrenti
Come calcolare la VRAM necessaria per un team

La VRAM totale necessaria è: VRAM modello + KV cache × utenti concorrenti. Il KV cache dipende dalla lunghezza del contesto attivo per utente — per sessioni chat tipiche (2K–8K token attivi) è contenuto.

Esempio: RTX 5090 32 GB + Qwen3.5-27B Q4

Calcolo VRAM
Modello Qwen3.5-27B Q4_K_M ~16 GB
KV cache per utente (8K context attivo) ~1.2 GB
5 utenti concorrenti × 1.2 GB ~6 GB
Overhead sistema ~1 GB
Totale stimato ~23 GB / 32 GB

Esempio: RTX 5070 Ti 16 GB + Qwen3.5-9B Q8

Calcolo VRAM
Modello Qwen3.5-9B Q8 ~9 GB
KV cache per utente (8K context attivo) ~0.7 GB
5 utenti concorrenti × 0.7 GB ~3.5 GB
Overhead sistema ~0.5 GB
Totale stimato ~13 GB / 16 GB
Attenzione: context window ≠ context attivo Il modello supporta 128K token di context window, ma il KV cache si accumula solo per i token effettivamente nella sessione. Una chat tipica ha 2K–8K token attivi, non 128K. Se le sessioni diventano molto lunghe (procedure lunghe + molti scambi), il KV cache cresce proporzionalmente.
